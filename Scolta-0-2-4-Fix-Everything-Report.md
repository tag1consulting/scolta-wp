## Bug 1 — WP-CLI Indexer Dispatch Regression

**Closed.** Hank's blocker is resolved.

### Root cause

The WP-CLI docblock for `[--indexer=<indexer>]` contained `default: auto`. WP-CLI treats `default:` as a machine directive, not documentation: it injects `$assoc_args['indexer'] = 'auto'` on every invocation that omits `--indexer`. Both `isset()` and `get_flag_value()` see the pre-injected value and can't distinguish it from an explicit flag. The admin setting was permanently unreachable. Commit `a286a93` (2026-04-17) replaced `get_flag_value` with `isset` but was a no-op — both helpers operate after injection.

### Fix

`b896f97` — Removed `default: auto` from the docblock. WP-CLI no longer injects when the flag is absent. The handler falls through to `$settings['indexer']`.

### Tests

- **`test_indexer_docblock_has_no_default_injection`**: source-parse test, extracts the `[--indexer]` docblock block and asserts `default:` is absent. Catches re-injection at the source.
- **`test_admin_php_setting_respected_without_cli_flag`**: handler called with empty `$assoc_args`; asserts 'php' from admin setting wins.
- **`test_injected_default_would_have_overridden_admin_setting`**: documents pre-fix behavior — when `$assoc_args['indexer'] = 'auto'` is injected, it overrides admin 'php'. Proves the bug and confirms the fix direction.
- **`tests/integration/test-indexer-dispatch.sh`** (`b405590`): spawns real `wp scolta build` against the DDEV site. With `indexer=php` in settings and no `--indexer` flag, asserts "Using PHP indexer pipeline" appears in output. 4/4 pass on `scolta-wordpress-7.ddev.site`.

### Documentation

- `tests/POSTMORTEM-dispatch-regression.md` — why the 2026-04-17 test matrix missed it (tests bypassed WP-CLI's argument parser by constructing `$assoc_args` directly)
- `tests/CONVENTIONS.md` — codified rule: integration tests for WP-CLI commands must exercise the argument parser, not just the handler in isolation

### Downstream verification

With `indexer=php` set in admin, `wp scolta build --force` produces:
```
Using PHP indexer pipeline.
Step 1: Gathering content...
...
PHP indexer complete: N pages indexed, M files written in Xs.
```
No `pagefind --site` invocation. No "Exported HTML files: 0" in admin status (panel correctly hides that row for PHP pipeline). Both downstream symptoms from Hank's report are gone.

### Commits / PR

`b896f97` (docblock fix), `b405590` (postmortem + conventions + integration test) — committed directly to `scolta-wp` main. CI: ✅

## Bug 2 — Drupal Search Results Page Not Using Full Width

**Closed.**

### Root cause

`packages/scolta-drupal/css/scolta.css` is generated by `composer copy-assets` from `packages/scolta-php/assets/css/scolta.css`. The Drupal copy was stale — it predated the scolta-php change that added the `.has-filters` conditional to the layout grid. The stale CSS had:

```css
.scolta-layout {
  grid-template-columns: 220px minmax(0, 1fr);  /* ← permanent two-column */
}
```

With no `.has-filters` modifier, the 220px filter sidebar was always present in the grid even when the `<aside>` was empty. On a single-site Drupal install, the filter column was always empty but always consumed 220px, squeezing all search results, AI summaries, and suggestion chips into the narrow right column. The `.scolta-filters:empty { display: none }` rule was also missing, so the empty aside was visually invisible but still occupied a grid track.

The scolta-php *source* was already correct (`.has-filters` conditional added in 0.2.3). The Drupal copy had not been regenerated.

### Fix

`59b2b69` on `fix/scolta-drupal-full-width-regression` — ran `composer copy-assets` to regenerate `css/scolta.css` from the correct scolta-php source. The generated file now has:

```css
.scolta-layout { grid-template-columns: 1fr; }
.scolta-layout.has-filters { grid-template-columns: 220px minmax(0, 1fr); }
.scolta-filters:empty { display: none; }
```

### Regression test

`tests/src/LayoutCssRegressionTest.php` — 4 tests:
1. Base `.scolta-layout` rule must be `1fr` (not hardcoded two-column)
2. `.scolta-layout.has-filters` must define the 220px column
3. `.scolta-filters:empty { display: none }` must be present
4. Responsive `700px` media query must target `.has-filters`, not the base rule

All 4 pass. CI runs `composer install` (which triggers `copy-assets`) before tests, ensuring the generated CSS is always current.

### Verification

On `scolta-drupal-11-nopf.ddev.site`: search results now fill the full content width. The AI Overview and suggestion chips are no longer squeezed.

WordPress unaffected — scolta-wp serves CSS from `vendor/tag1/scolta-php/assets/css/scolta.css` directly, which was already correct.

https://github.com/tag1consulting/scolta-drupal/pull/2 — CI: ✅
